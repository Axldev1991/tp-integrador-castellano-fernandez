# 🎮 Controllers

Los **Controllers** constituyen el corazón de la aplicación.

Toda la lógica de negocio del sistema se encuentra implementada dentro de ellos.

Un controlador recibe una petición HTTP, procesa la información recibida, interactúa con los modelos y finalmente devuelve una respuesta al cliente.

Los controladores **no realizan consultas SQL directamente**, sino que delegan esa responsabilidad a los Models.

---

# 📚 ¿Qué responsabilidades tiene un Controller?

Un controlador puede encargarse de:

- Leer datos enviados por el cliente.
- Obtener parámetros de la URL.
- Validar información (cuando no corresponde a un middleware).
- Llamar a uno o varios modelos.
- Procesar resultados.
- Manejar errores.
- Renderizar vistas EJS.
- Enviar respuestas JSON.
- Redireccionar al usuario.

---

# postLogin()

## Responsabilidad

Autenticar un administrador.

---

## Método HTTP

POST

---

## Datos recibidos

```text

contraseña
```

---

## Proceso

1. Obtiene los datos enviados desde el formulario.
2. Busca el usuario correspondiente.
3. Compara la contraseña ingresada.
4. Si es correcta crea una sesión.
5. Registra el acceso en la tabla de auditoría.
6. Redirecciona al Dashboard.

---

Vuelve a mostrar el Login indicando el problema.

---

## Flujo

```text
Formulario

↓

postLogin()

↓

Buscar Usuario

↓

Comparar Contraseña

↓

Crear Session

↓

Registrar Auditoría

↓

Dashboard
```

# postVenta()

Es una de las funciones más importantes del proyecto.

---

## Responsabilidad

Registrar una nueva venta.

---

## Información recibida

```json
{
    "cliente":"Juan",

    "total":25000,

    "items":[]
}
```

---

## Proceso

1. Verifica los datos recibidos.
2. Inicia una transacción SQL.
3. Inserta la venta.
4. Inserta cada producto vendido.
5. Confirma la transacción.
6. Devuelve éxito.

---

## ¿Por qué utilizar una transacción?

Supongamos una venta con cinco productos.

Si el cuarto producto produce un error:

Sin transacciones:

```text
Producto 1 ✔

Producto 2 ✔

Producto 3 ✔

Producto 4 ✖

Producto 5 ❌
```

La venta quedaría incompleta.

---

Con transacciones:

```text
Error

↓

ROLLBACK

↓

No se guarda absolutamente nada.
```

Esto garantiza la integridad de los datos.

---

## Respuesta

```json
{
    "status":"success",
    "message":"Venta registrada correctamente."
}
```

# 🧩 Manejo de Errores

Todos los controladores siguen una estructura similar.

```javascript
try{

    // lógica

}
catch(error){

    // responder error

}
```

Esto evita que una excepción provoque la caída completa del servidor.

En caso de error siempre se responde con un código HTTP apropiado y un mensaje descriptivo.

---

# 📌 Buenas Prácticas Aplicadas

Durante el desarrollo de los controladores se siguieron diversas buenas prácticas:

- Separación entre lógica de negocio y acceso a datos.
- Uso de funciones asíncronas (`async/await`).
- Delegación de consultas SQL a los modelos.
- Validación previa mediante middlewares.
- Manejo centralizado de errores.
- Respuestas consistentes para el frontend.
- Reutilización de funciones.

Gracias a esta organización, los controladores son más fáciles de mantener, probar y extender.
