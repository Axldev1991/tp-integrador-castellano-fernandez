# 🗄️ Models

Los **Models** son la capa encargada de comunicarse directamente con la base de datos.

Toda consulta SQL del proyecto se encuentra centralizada en esta capa.

Gracias a esta arquitectura, los controladores no necesitan conocer cómo funciona la base de datos; simplemente invocan funciones del modelo y reciben la información procesada.

---

# 📚 ¿Qué responsabilidades tiene un Model?

Un Model tiene como objetivo:

- Ejecutar consultas SQL.
- Insertar registros.
- Actualizar información.
- Eliminar registros (cuando corresponde).
- Obtener datos de una o varias tablas.
- Ejecutar transacciones.
- Devolver resultados al controlador.

Los Models **no renderizan vistas** ni responden peticiones HTTP.

Su única responsabilidad consiste en acceder a los datos.

# getUsuarioByCorreo()

## Responsabilidad

Buscar un administrador mediante su correo electrónico.

---

## Parámetro

```text
correo
```

---

## Consulta SQL

Internamente realiza una consulta similar a:

```sql
SELECT *
FROM administradores
WHERE correo = ?;
```

## Uso

Es utilizada durante el proceso de Login para verificar si el usuario existe.

- correo

# 🔄 Transacciones

Las transacciones son fundamentales en este proyecto.

Especialmente durante el registro de ventas.

---

## ¿Qué es una transacción?

Es un conjunto de consultas SQL que deben ejecutarse completamente o no ejecutarse en absoluto.

---

## Flujo

```text
BEGIN

↓

INSERT Venta

↓

INSERT Producto 1

↓

INSERT Producto 2

↓

INSERT Producto 3

↓

COMMIT
```

---

Si ocurre un error

```text
BEGIN

↓

INSERT Venta

↓

INSERT Producto 1

↓

ERROR

↓

ROLLBACK
```

No queda información inconsistente en la base de datos.

---

# 📚 ¿Por qué separar Controllers y Models?

Supongamos que un controlador ejecutara directamente SQL.

```javascript
SELECT * FROM productos;
```

Si mañana cambia la estructura de la base de datos habría que modificar decenas de controladores.

Con el patrón MVC únicamente cambia el Model.

Los Controllers continúan funcionando exactamente igual.

Esta separación mejora:

- Mantenimiento.
- Reutilización.
- Organización.
- Escalabilidad.
- Pruebas unitarias.

# 📌 Buenas prácticas implementadas

Los modelos del proyecto siguen varias buenas prácticas:

- Una única responsabilidad por función.
- Uso de consultas parametrizadas para evitar SQL Injection.
- Reutilización de funciones.
- Separación entre lógica de negocio y acceso a datos.
- Uso de transacciones cuando la operación afecta múltiples tablas.
- Devolución de datos estructurados al controlador.

# 🔗 Relación entre todas las capas

El siguiente diagrama resume la arquitectura completa del proyecto:

```text
                 Cliente
                    │
                    ▼
               Express Router
                    │
                    ▼
              Middlewares
                    │
                    ▼
               Controllers
                    │
                    ▼
                 Models
                    │
                    ▼
             MySQL / MariaDB
                    │
                    ▲
                 Resultados
                    │
                    ▲
               Controllers
                    │
                    ▼
          JSON / Vista EJS
                    │
                    ▼
                 Cliente
```

Este flujo representa el funcionamiento de prácticamente todas las operaciones del sistema, desde la autenticación de un administrador hasta el registro de una venta o la consulta de productos.
