# ⚙️ Configuración e Inicialización del Backend

Esta sección documenta cómo se inicializa el servidor, cómo se conecta con la base de datos y cuál es la función de cada archivo relacionado con la configuración del proyecto.

---

# 📂 Estructura Inicial del Backend

```text
backend/
│
├── index.js
│
├── package.json
│
├── .env
│
└── src/
    │
    ├── config/
    │      └── environments.js
    │
    ├── database/
    │      ├── db.js
    │      ├── schema.sql
    │      └── seed.js
    │
    ├── routes/

```

Todos estos archivos trabajan juntos para iniciar el servidor y preparar el entorno antes de comenzar a recibir peticiones HTTP.

---

# 🚀 index.js

## ¿Qué es?

Es el punto de entrada del servidor.

Cuando ejecutamos:

```bash
npm run dev
```

o

```bash
node index.js
```

Node.js comienza la ejecución exactamente desde este archivo.

Toda aplicación Express necesita un punto de inicio.

---

## Responsabilidades

El archivo `index.js` tiene varias responsabilidades importantes.

### 1. Crear el servidor Express

Lo primero que hace es crear una instancia de Express.

```javascript
const app = express();
```

A partir de ese momento existe un servidor HTTP listo para configurarse.

---

### 2. Configurar Middlewares Globales

Antes de definir cualquier ruta se configuran distintos middlewares.

Por ejemplo:

- Lectura de JSON

```javascript
app.use(express.json());
```

Permite recibir:

```json
{
    "nombre":"Remera"
}
```

---

Lectura de formularios HTML

```javascript
app.use(express.urlencoded({ extended:true }));
```

Permite leer formularios enviados desde EJS.

---

Archivos estáticos

```javascript
app.use(express.static("public"));
```

Hace públicos:

- CSS
- JavaScript
- Imágenes
- Iconos
- Archivos subidos

Sin esto el navegador no podría cargar ningún recurso visual.

---

Sesiones

Express Session guarda información del usuario entre una petición y otra.

Gracias a las sesiones un administrador puede iniciar sesión una única vez y navegar por todo el panel sin volver a autenticarse en cada página.

---

Motor de vistas

```javascript
app.set("view engine","ejs");
```

Le indica a Express que utilizará EJS para renderizar páginas HTML dinámicas.

---

### 3. Registrar las rutas

Luego Express conecta todas las rutas del proyecto.

Ejemplo:

```javascript
app.use("/admin",adminRoutes);
```

Esto significa que todas las rutas definidas dentro de:

```
admin.routes.js
```

comenzarán con

```
/admin
```

Por ejemplo:

```
GET /dashboard
```

pasa a convertirse en

```
GET /admin/dashboard
```

---

### 4. Iniciar el servidor

Finalmente Express comienza a escuchar conexiones.

```javascript
app.listen(PORT)
```

Desde ese momento cualquier navegador puede comunicarse con el servidor.

---

# 🌎 Variables de Entorno

El proyecto utiliza un archivo `.env`.

Su objetivo es almacenar información sensible sin escribirla directamente en el código.

Ejemplo:

```env
PORT=3000

DB_HOST=localhost

DB_USER=root

DB_PASSWORD=1234

DB_NAME=tp_autoservicio

SESSION_SECRET=mi_clave_super_secreta
```

---

## ¿Por qué utilizar un .env?

Porque evita subir información privada a GitHub.

Si las credenciales estuvieran escritas directamente en el código cualquier persona podría acceder a ellas.

Además permite cambiar configuraciones sin modificar el programa.

Por ejemplo:

En desarrollo

```env
PORT=3000
```

En producción

```env
PORT=80
```

Sin cambiar una sola línea de JavaScript.

---

# 📂 config/

La carpeta **config** contiene toda la configuración del proyecto.

Actualmente posee el archivo:

```
environments.js
```

---

# 📄 environments.js

## Responsabilidad

Leer las variables del archivo `.env` y convertirlas en un objeto que pueda utilizar el resto del sistema.

En lugar de escribir constantemente:

```javascript
process.env.DB_HOST
```

se utiliza:

```javascript
environments.database.host
```

Esto hace que el código sea mucho más limpio y organizado.

---

## Información que administra

Generalmente contiene:

- Puerto del servidor
- Host de MySQL
- Usuario
- Contraseña
- Nombre de la base
- Puerto de MySQL
- Secret de Session

---

# 🗄 database/

Esta carpeta contiene todo lo relacionado con la base de datos.

Su objetivo es mantener separada la lógica SQL del resto del proyecto.

Está compuesta por:

```text
database/

db.js

schema.sql

seed.js
```

---

# 📄 db.js

## Responsabilidad

Crear la conexión con MySQL.

Todos los modelos utilizan esta conexión.

Nunca crean una nueva.

---

## ¿Por qué?

Crear una conexión nueva para cada consulta sería extremadamente lento.

Por eso se utiliza un **Connection Pool**.

---

# ¿Qué es un Connection Pool?

Un Pool es un conjunto de conexiones abiertas.

En lugar de hacer esto:

Cliente

↓

Abrir conexión

↓

Consulta

↓

Cerrar conexión

↓

Abrir nuevamente

↓

Consulta

↓

Cerrar

El Pool mantiene varias conexiones abiertas permanentemente.

```
Pool

Conexión 1

Conexión 2

Conexión 3

Conexión 4

Conexión 5
```

Cada petición toma una conexión libre.

Cuando termina la devuelve al Pool.

Esto mejora muchísimo el rendimiento.

---

## Configuración del Pool

Entre las configuraciones más importantes encontramos:

### host

Dirección del servidor MySQL.

---

### user

Usuario de la base.

---

### password

Contraseña.

---

### database

Nombre de la base.

---

### waitForConnections

Si todas las conexiones están ocupadas espera hasta que una quede libre.

---

### connectionLimit

Cantidad máxima de conexiones abiertas simultáneamente.

En este proyecto:

```javascript
connectionLimit:10
```

Hasta diez consultas pueden ejecutarse al mismo tiempo.

---

### queueLimit

Cantidad máxima de consultas esperando.

```javascript
queueLimit:0
```

Significa que no existe un límite.

---

# 📄 schema.sql

## ¿Qué es?

Es el script SQL encargado de crear toda la estructura de la base de datos.

No contiene información.

Solo crea:

- Tablas
- Relaciones
- Claves primarias
- Claves foráneas
- Índices
- Restricciones

---

## ¿Por qué es importante?

Gracias a este archivo cualquier persona puede reconstruir completamente la base de datos.

Solo debe ejecutar:

```bash
mysql -u root -p tp_autoservicio < schema.sql
```

Y automáticamente se crearán todas las tablas necesarias.

---

# 📄 seed.js

## ¿Qué es?

Un Seed es un script que inserta información inicial.

Generalmente:

- Usuarios administradores
- Productos de ejemplo
- Configuraciones iniciales

---

## ¿Por qué existe?

Cuando otra persona descarga el proyecto la base de datos está vacía.

El Seed permite poblarla automáticamente.

En este proyecto crea el usuario administrador inicial para poder ingresar al sistema.

---

## Ejecución

```bash
node seed.js
```

Una vez ejecutado el sistema ya posee un administrador listo para iniciar sesión.

---

# 🔄 Ciclo de Vida del Servidor

Cada vez que se inicia el proyecto ocurre exactamente el siguiente proceso:

```text
Node.js

↓

Lee index.js

↓

Carga variables .env

↓

Carga environments.js

↓

Crea Express

↓

Configura Middlewares

↓

Configura Session

↓

Configura EJS

↓

Registra Rutas

↓

Conecta MySQL

↓

Comienza a escuchar conexiones

↓

Servidor listo
```

A partir de este momento el backend queda esperando solicitudes HTTP provenientes del frontend o del panel administrativo.

# 📌 Resumen

En esta sección se documentó toda la infraestructura necesaria para iniciar el sistema.

Los archivos analizados cumplen las siguientes funciones:

| Archivo | Responsabilidad |
|----------|-----------------|
| index.js | Inicializa el servidor Express y registra toda la configuración global. |
| environments.js | Centraliza las variables de entorno del proyecto. |
| db.js | Crea el Pool de conexiones hacia MySQL/MariaDB. |
| schema.sql | Define la estructura completa de la base de datos. |
| seed.js | Inserta información inicial necesaria para comenzar a utilizar el sistema. |

Con estos componentes correctamente configurados, el backend queda preparado para comenzar a recibir solicitudes HTTP y ejecutar la lógica de negocio del proyecto.
